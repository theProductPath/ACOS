## Framework docs

### Fill in `docs/adopting-acos.md`

Currently a skeleton at [`Products/ACOS/docs/adopting-acos.md`](../docs/adopting-acos.md). It's the guide for a new company adopting ACOS — what to set up, what to copy from `framework/templates/`, what conventions to expect.

- **Why it matters:** Without this, every future adopter has to reverse-engineer adoption from tPPOS. With it, ACOS becomes legitimately reusable.
- **First step:** Walk through the tPPOS instance step-by-step as if you were setting it up from scratch, and write each step down. The result is a v1 adoption guide grounded in a real worked example.
- **Effort:** Medium. The skeleton already lists the right sections.

### Common-task cheat sheet

"For task type X, read these folders." Was flagged as a TODO in the pre-extraction tPPOS README. The framework version names task types abstractly (writing a proposal, processing a transcript, scaffolding a new client) and maps them to which README/brief/manifest to read in what order.

- **Why it matters:** Gives agents a fast path to context without having to walk the full cascade for routine tasks.
- **First step:** Identify the 5-7 most common task types that show up in tPPOS work, then write the read-order for each. Land in `framework/README.md` as a new section.
- **Effort:** Small to medium.

## Framework conventions

### Decide on `brief-product.md` template

Flagged as TODO in the pre-extraction tPPOS README. Question: do products warrant their own brief template parallel to `brief-company` and `brief-client`, or do they use `folder-readme-item` plus an ad-hoc README?

- **Why it matters:** Products are a first-class folder type in ACOS. If they need a canonical identity record (mission, positioning, target users, success metrics), a template stabilizes that.
- **First step:** Look at three or four existing product READMEs in `tPPOS Products/` (LAIR, agentcomms, candidate-interview-kit). What's recurring? If a clear shape emerges, write the template. If not, defer with a note.
- **Effort:** Small if the shape is obvious; medium if it requires real synthesis.

### Machine-readable manifest format

Should ACOS define a top-level `acos.yaml` or `manifest.json` that tools can parse structurally — listing folders, types, owners, last-updated dates — without walking READMEs?

- **Why it matters:** Markdown is human-friendly but tool-hostile for "show me every stale README" kinds of queries. A manifest makes the system programmatically inspectable.
- **First step:** Decide if the use case is real. Is anyone going to write the tool that consumes this manifest? If yes, prototype the schema. If no, leave the question open — the frontmatter `type` taxonomy already gives partial structure.
- **Effort:** Medium. The schema is straightforward; the question is whether to build it before there's a consumer.

### Pattern set — fifth README pattern?

The four README patterns today (root / container / item / asset) cover everything tPPOS needs. Watch for cases that don't fit; if a fifth pattern emerges from real work, add it.

- **Why it matters:** Adding a pattern is a real commitment — every instance has to learn the new vocabulary. Don't add one casually.
- **First step:** None until a real case appears. Note here when one does.
- **Effort:** N/A — passive watch.

## Skill library

### Promote additional skills from tPPOS

`client-brief-processor` is the first ACOS skill. Other skills currently live as plugins or user-level skills outside tPPOS but operate against tPP context: `candidate-assessment`, `app-assessment`, `tpp-doc`, `new-offering`. Each is a promotion candidate if it would genericize cleanly.

- **Why it matters:** A richer skill library is what makes ACOS legitimately valuable to a second adopter. The framework's promise is "drop in these skills, configure via overlays, get to work."
- **First step:** Pick one (suggestion: `app-assessment` is high-value and reads the most generic; `candidate-assessment` is also strong). Apply the framework-vs-instance rule from [`docs/extending-acos.md`](../docs/extending-acos.md): extract the generic version into `framework/skills/`, scrub instance specifics, leave an overlay in tPPOS for tPP-specific configuration.
- **Effort:** Medium per skill. Each promotion uses the same playbook as today's client-brief-processor extraction.

### Future input sources for client-brief-processor

The current skill names three deferred input sources: Notion transcripts, audio/video transcripts, Slack exports. Each is implementable at the framework level (with the actual integration via instance overlay).

- **Why it matters:** Today the skill is only useful for pasted transcripts and email (the latter only with overlay support). Wiring up Notion alone would unlock a much wider set of inputs for tPP and any future adopter using Notion.
- **First step:** Decide which input source to add first. If Notion, write the generic Notion-database-read protocol in the SKILL.md (under "Input sources"), and in tPPOS add a Notion DB ID to the overlay.
- **Effort:** Medium per source.

## Product-level

### Extract ACOS into its own repo

Per the `Products/` convention ("each product folder is a self-contained GitHub repository"), ACOS will eventually move out of the tPPOS tree into its own repo. Defer until the surface stabilizes — likely after a second adopter exists or the framework hits v1.0.

- **Why it matters:** Final state per the Products convention. Until then, ACOS lives inside the tPP tree but is logically separate.
- **First step:** Not yet. Add to this list to revisit at v0.5 or when a second instance materializes.
- **Effort:** Small (mechanics) but coordination-heavy (link updates across tPPOS).

### Adoption tooling — scaffold-a-new-instance script

A small CLI or skill that scaffolds a new ACOS instance: creates the folder structure, copies templates into place, prompts for company name to fill placeholders. Out of scope now but worth noting.

- **Why it matters:** Lowers the barrier to a second adopter. Each manual step in adoption is friction.
- **First step:** None now. Defer until adoption guide is written and a second adopter is real.
- **Effort:** Medium.
